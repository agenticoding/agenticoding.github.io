סיימת את המימוש. הבדיקות עוברות. הסוכן ביצע את התוכנית שלך בהצלחה. עכשיו מגיעה השאלה הקריטית: האם זה באמת נכון?

זהו שלב ה**אימות** מ[תהליך העבודה בארבעה שלבים של שיעור 3](../methodology/lesson-3-high-level-methodology.md)—"שער האיכות" השיטתי לפני דילוור. סקירת קוד (code review) תופסת את השגיאות ההסתברותיות שסוכנים בהכרח מכניסים: באגי לוגיקה עדינים, אי-התאמות ארכיטקטוניות, מקרי קצה שטופלו בצורה שגויה, דפוסים שלא לגמרי מתאימים לבסיס הקוד שלך.

תובנת המַפְתח: **סקור בקונטקסט טרי, נפרד מקונטקסט כתיבת הקוד.** זה מונע הטיית אישור (confirmation bias) וממנף את האופי הסטייטלס של סוכנים מ[שיעור 1](../fundamentals/lesson-1-how-llms-work.md) ו[שיעור 2](../fundamentals/lesson-2-how-agents-work.md). סוכן שסוקר את העבודה שלו באותה שיחה יגן על ההחלטות שלו. סוכן בקונטקסט טרי ינתח באובייקטיביות, ללא היצמדות לבחירות קודמות.

:::info בסיסי קוד המנוהלים אך ורק על ידי סוכנים לעומת בסיסי קוד מעורבים: הבחנה קריטית

אותם תקני הנדסה—DRY, YAGNI, ארכיטקטורה, תחזוקתיות, קריאות—חלים על כל בסיסי הקוד. מה שמשתנה הוא אופטימיזציית סגנון הקוד ותהליך הסקירה:

**בסיסי קוד המתוחזקים ע"י סוכן בלבד** מתוחזקים אך ורק על ידי AI עם התערבות אנושית מינימלית ברמת הקוד. העדיפו סגנון קוד מעט מוטה עבור בהירות לכלי AI: יותר type annotations מפורשים, תיעוד קצת יותר מפורט, קבצי קונטקסט ארכיטקטוניים מפורטים ([שיעור 6](./lesson-6-project-onboarding.md)). שאלת ריוויו: "האם סוכן יבין את זה בעוד 6 חודשים?"

**בסיסי קוד מעורבים** מאזנים שיתוף פעולה אנושי ו-AI שבו שניהם עובדים ישירות עם קוד. העדיפו סגנון קוד שמתאים לתמציתיות אנושית תוך שמירה על ניווטיות והתמצאות קלה לAI. **רוב בסיסי הקוד בפרודקשן שייכים לקטגוריה זו.**

**הבדל קריטי בבסיסי קוד מעורבים:** הוסיפו שלב סקירה ידני שבו אתם קוראים ובודקים במלואו קוד שנוצר על ידי AI לפני קומיט כדי להבטיח קריאוּת אנושית. זה לא נתון לוויכוח—ללא כללי פרויקט מפורשים שמנחים סגנון, סוכנים מייצרים קוד העוקב אחר דפוסים מנתוני האימון שלהם שעשויים לא להתאים לתקני הקריאוּת של הצוות שלכם. כוונו את כללי הפרויקט שלכם ([שיעור 6](./lesson-6-project-onboarding.md)) להנחות סוכנים לסגנון כתיבה שבני אדם מצפים לו, ואז לאמת שהפלט עומד בציפיות אלה.

:::

## תבנית הפרומפט לקוד ריוויו וסקירה כללית

תבנית זו משלבת טכניקות מ[שיעור 4: Prompting 101](../methodology/lesson-4-prompting-101.md). הבנת **למה** כל אלמנט קיים מאפשרת לך להתאים דפוס זה למשימות סקירה אחרות (ביקורות אבטחה, ניתוח ביצועים, סקירה ארכיטקטונית).

```markdown
You are an expert code reviewer. Analyze the current changeset and provide a critical review.

The changes in the working tree were meant to: $DESCRIBE_CHANGES

Think step-by-step through each aspect below, focusing solely on the changes in the working tree.

1. **Architecture & Design**
   - Verify conformance to project architecture
   - Check module responsibilities are respected
   - Ensure changes align with the original intent

2. **Code Quality**
   - Code must be self-explanatory and readable
   - Style must match surrounding code patterns
   - Changes must be minimal - nothing unneeded
   - Follow KISS principle

3. **Maintainability**
   - Optimize for future LLM agents working on the codebase
   - Ensure intent is clear and unambiguous
   - Verify comments and docs remain in sync with code

4. **User Experience**
   - Identify areas where extra effort would significantly improve UX
   - Balance simplicity with meaningful enhancements

Review the changes critically. Focus on issues that matter.
Use ChunkHound's code research.
DO NOT EDIT ANYTHING - only review.
```

אחרי כתיבת קוד ([שיעור 7](./lesson-7-planning-execution.md)), כתיבת בדיקות ([שיעור 8](./lesson-8-tests-as-guardrails.md)), ולאחר שכל הבדיקות עוברות, שלב סקירה זה תופס מה שתהליך הפיתוח האיטרטיבי השאיר מאחור—שער האיכות האחרון לפני קומיט.

:::tip הפניה
ראה את תבנית הפרומפט המלאה עם הנחיות סקירה איטרטיבית: [Comprehensive Code Review](/prompts/code-review/comprehensive-review)
:::

### סקירה איטרטיבית: חזור עד ש"ירוק" או עד שהערך יורד

סקירת קוד היא לעתים רחוקות מעבר אחד—סקירה ראשונה מוצאת בעיות, מתקנים אותן, מריצים בדיקות מחדש ([שיעור 8](./lesson-8-tests-as-guardrails.md)) לתפיסת רגרסיות, ואז סוקרים שוב בקונטקסט טרי (לא אותה שיחה שבה הסוכן יגן על החלטותיו הקודמות). המשיכו במחזור זה: סקירה בקונטקסט טרי, תיקון בעיות, אימות עם בדיקות, חזרה.

**הסקירה עצמה היא הסתברותית**— גם היא מבוססת LLM שעושה תחזיות סטטיסטיות. הסוכן יכול לטעות. הוא עשוי להציע "תיקונים" ששוברים קוד עובד או מכניסים רגרסיות שסט הבדיקות שלך תופס.

כאן שיקול הדעת שלך כמפעיל|ה הופך חיוני (ה"אמנות" של התהליך):

- **בדיקות עוברות + סקירה ירוקה** = לדלוור
- **בדיקות עוברות + סקירה שמחזירה רק הערות קטנוניות (nitpicking)** = לדלוור
- **בדיקות נכשלות אחרי "תיקוני" סקירה** = הסקירה כנראה טעתה, יש לדחות את ההצעה

הפסיקו את האיטרציות כשמגיעים ל**אור ירוק** (אין בעיות מהותיות, בדיקות עוברות) או *ירידה בערך הסקירה**. ירידת ערך מתבטאת כ:

- **ניטפיקינג**: הערות קטנוניות,  העדפות סגנון טריוויאליות כמו "שנה שם למשתנה הזה"
- **הזיות**: הסוכן ממציא בעיות לא קיימות או מציע דפוסים שלא מתאימים לארכיטקטורה שלך
- **נפילות טסטים שנגרמו מסקירה**: ה"תיקון" שבר קוד שעבד קודם
- **עלות מופרזת**: 4+ איטרציות לבעיות קטנות שנותרו

בנקודה זו, סמכו על הטסטים שלכם כפוסק האובייקטיבי ודלוורו את הקוד—סקירת AI נוספת עולה יותר מהערך שהיא נותנת ועלולה להוביל להדרדרות באיכות.

## Pull Requests לסוקרים אנושיים ו-AI

Pull requests משרתים שני קהלים: מתחזקים אנושיים ועוזרי סקירת ה-AI שלהם. קהלים אלה מעבדים מידע באופן שונה מהותית:

- **סוקרים אנושיים** סורקים מהר, מסיקים משמעות מקונטקסט, ומעדיפים סיכומים תמציתיים (1-3 פסקאות מקסימום). הם רוצים להבין את ה"למה" ואת הערך העסקי במבט חטוף.

- **עוזרי סקירת AI** מעבדים תוכן חלק-אחר-חלק, מתקשים עם כינויים מעורפלים וסחף סמנטי (semantic drift), וצריכים מבנה מפורש ([שיעור 5](../methodology/lesson-5-grounding.md)). הם דורשים קונטקסט טכני מפורט: שינויי קבצים ספציפיים, דפוסים ארכיטקטוניים, שינויים-שוברים המפורטים בבירור.

תיאורי PR מסורתיים מותאמים לקהל אחד או לאחר — הם עלולים להיות מילוליים מדי לבני אדם או מעורפלים מדי לסוכני AI. הפתרון: ליצור את שניהם בתהליך עבודה מתואם באמצעות תת-סוכנים (sub-agents).

### מנגנונים בפעולה

**תת-סוכנים לשימור קונטקסט ([שיעור 5](../methodology/lesson-5-grounding.md#solution-2-sub-agents-for-context-isolation)):**

ההוראה "Using the sub task tool to conserve context" מייצרת סוכן נפרד לחקירת היסטוריית git, ומונעת מהקונטקסט של המתזמר הראשי להתמלא ב-diff של קומיטים. תת-הסוכן מחזיר רק ממצאים מסונתזים. בלי זה, חקירת 20-30 קבצים שהשתנו צורכת 40K+ טוקנים, דוחפת אילוצים קריטיים לאמצע הנשכח של עקומת הקשב עם צורת הU.

יכולת תת-סוכנים זו ייחודית ל-[Claude Code CLI](/developer-tools/cli-coding-agents#claude-code). כלים אחרים (Codex, GitHub Copilot) דורשים פיצול זה לפרומפטים רציפים מרובים: חקור קודם, ואז נסח על סמך ממצאים.

**עיגון מרובה מקורות (multi-source grounding) ([שיעור 5](../methodology/lesson-5-grounding.md#production-pattern-multi-source-grounding)):** ArguSeek חוקר שיטות עבודה מומלצות ל-PR בעוד ChunkHound מעגן את תיאור הPR לארכיטקטורה ולסגנון הקידוד של בסיס הקוד שלך.

**Prompting מובנה ([שיעור 4](../methodology/lesson-4-prompting-101.md)):** פרסונה, אילוצי תקשורת, גבולות פורמט, ודרישות מבניות מכוונים את הסוכן לייצר פלטים מותאמים כפולים (לבן אנוש וסוכן).

**דרישת ראיות ([שיעור 7](./lesson-7-planning-execution.md#require-evidence-to-force-grounding)):** הפרומפט מאלץ עיגון דרך "explore the changes" ו-"learn the architecture"—הסוכן לא יכול לנסח תיאורים מדויקים מבלי לקרוא קומיטים וקוד אמיתיים.

### סקירת PRs עם עוזרי AI

כשנמצאים בצד המקבל של PR עם תיאורים מותאמים כפולים, יש לנו קונטקסט מובנה הן להבנה אנושית והן לסקירה בסיוע AI. סעיף זה מראה איך למנף את שני התיאורים ביעילות.

#### צריכת התיאורים הכפולים

**התיאור המותאם לאדם** (תיאור PR ב-GitHub):

- לקרוא ראשית להבנת ה"למה" וערך עסקי
- לסרוק מהר למציאת שינויים-שוברים וקבצי-מפתח שהושפעו
- להשתמש בזה ליצירת המודל המנטלי הראשוני שלך של השינויים

**התיאור המותאם ל-AI** (`PR_REVIEW_CONTEXT.md` או דומה):

- להזין לעוזר סקירת ה-AI שלכם (GitHub Copilot, Codex, Claude Code, וכו')
- מספק קונטקסט טכני מקיף שה-AI צריך כדי לבצע ניתוח מדויק
- מכיל טרמינולוגיה מפורשת, נתיבי קבצים, ודפוסים ארכיטקטוניים

#### דפוס הפרומפט לסקירה

כשסוקרים PR עם תיאורים מותאמים כפולים, כדאי להשתמש בדפוס זה עם עוזר ה-AI שלכם:

:::tip Chain of Draft (CoD): חלופה יעילה ל-Chain of Thought

**שימו לב לטכניקה בצעד 2 של תהליך הסקירה למעלה:**

> <div dir='ltr'>"Think step by step, but only keep a minimum draft for each thinking step, with 5 words at most. Return the assessment at the end of the response after a separator ####."</div>

זה **שרשרת טיוטות (CoD)**—אופטימיזציה של פרומט מבוסס שרשרת חשיבה (CoT) שמקיימת היגיון מובנה תוך יעילות רבה יותר.

**איך זה עובד:** במקום לייצר הסברים מפורטים שלב-אחר-שלב, CoD מורה ל-LLM לחשוב דרך כל שלב אבל לשמור את הטיוטה תמציתית (5 מילים מקסימום לשלב), ואז להחזיר את ההערכה הסופית אחרי מפריד (`####`).

**למה להשתמש בזה לסקירות:** CoD מספק את אותם יתרונות שימוש-בהיגיון כמו CoT—פירוק ניתוח מורכב לשלבים לוגיים—אבל עם צריכת טוקנים מופחתת וזמני תגובה מהירים יותר.

**ראו עוד:**

- [מאמר המחקר המקורי](https://arxiv.org/abs/2502.18600)
- [תיעוד מתוך האתר Learn Prompting](https://learnprompting.org/docs/advanced/thought_generation/chain-of-draft)

:::

## נקודות מפתח

- **סקירה בקונטקסט טרי למניעת הטיית אישור** - סוכן שסוקר את עבודתו באותה שיחה יגן על החלטותיו. קונטקסט טרי מספק ניתוח אובייקטיבי.

- **החלת אותה מתודולוגית ארבעה שלבים לסקירה** - מחקר (הבן כוונה) ← תכנון (מבנה סקירה) ← ביצוע (בצע ניתוח) ← אימות (החלט לדלוור, לתקן, או לייצר מחדש).

- **שימוש בפרומפטים סקירה מובנים עם Chain-of-Thought** - תבנית הסקירה משיעור זה מחילה עקרונות prompting של שיעור 4: פרסונה, CoT, מבנה, עיגון, אילוצים. התאימו דפוס זה לסקירות אבטחה, ניתוח ביצועים, או אימות ארכיטקטוני.

- **איטרציה עד אור ירוק או ערך יורד** - תקנו בעיות, ואז בצעו סקירה מחדש בקונטקסט טרי. עיצרו כשהממצאים הופכים לניטפיקינג טריוויאלי או שהסוכן הוזה בעיות שלא קיימות.

- **דרישת ראיות מאלצת עיגון** - "Provide file paths and line numbers" מ[שיעור 7](./lesson-7-planning-execution.md#require-evidence-to-force-grounding) מבטיח שממצאי הסקירה מבוססים על קוד אמיתי, לא ניחושים סטטיסטיים.

- **כיתבו תיאורי PR כפולים מותאמים לסוקרים אנושיים ו-AI** - בני אדם צריכים סיכומים תמציתיים וניתנים לסריקה (1-3 פסקאות). עוזרי AI צריכים קונטקסט טכני מקיף וחד-משמעי. צרו את שניהם בתהליך עבודה מתואם על מנת לשרת את שני הקהלים ביעילות.

- **מינוף קונטקסט מותאם ל-AI כשסוכן מבצע את הסקירה PRs** - כשסוכן סוקר PR עם תיאורים כפולים, הזן את התיאור המותאם ל-AI לעוזר הסקירה שלך. זה מספק את העיגון והקונטקסט הארכיטקטוני הנדרשים לניתוח מדויק, מפחית הזיות ומשפר את איכות סקירה.
