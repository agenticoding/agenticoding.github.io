[שיעור 5: עיגון](../methodology/lesson-5-grounding.md) כיסה איך RAG וחיפוש סמנטי מאפשרים לסוכנים לאחזר קונטקסט מקוד הבסיס שלכם ומהרשת. שיעור זה מתמקד במה שקורה במהלך תכנון וביצוע—איך לעגן באופן פעיל את הסוכן בזמן שאתם עובדים, איך לסקור את התכנון שלו לפני ביצוע אוטונומי, ואיך להגדיר workflows שתומכים בפיתוח מקבילי על פני מספר מופעים של סוכנים.

המעבר מאיסוף קונטקסט לשימוש בקונטקסט הוא קריטי. עיגון אינו פעילות חד-פעמית מראש—הוא רציף. אתם טוענים קונטקסט, סוקרים את תוכנית הסוכן, נותנים לו לבצע, ואז מאמתים. כשמשהו לא מתאים למודל המנטלי שלכם, אתם עוצרים ומבהירים. כשהסוכן מציע שכפול, אתם אוכפים DRY. שיעור זה מכסה את הטכניקות הטקטיות שהופכות סוכנים ממחוללי קוד למכונות יצור קוד אמינות.

## טיפים לעיגון: הנדסת קונטקסט פעילה

ביצוע אפקטיבי דורש ניהול קונטקסט מכוון. טכניקות אלה שומרות את הסוכן מעוגן בקוד הבסיס האמיתי שלכם, לא בדפוסים סטטיסטיים מנתוני אימון.

### תמיד לעגן בקוד

הראו לסוכנים דפוסים אמיתיים מקוד הבסיס שלכם, לא תיעוד גנרי. תיאורים מופשטים מובילים לפתרונות גנריים שלא מתאימים לקוד הבסיס הספציפי שלכם.

**דוגמה:** אתם צריכים להוסיף rate limiting ל-API שלכם. אל תכתבו פרומפט כמו "Add rate limiting middleware." במקום זאת השתמשו בפרומפט שמעגן את הסוכן לקוד שלכם:
<div dir='ltr'>
"Search for existing middleware patterns, especially authentication. Check our Redis configuration.  
Then propose rate limiting that follows the same error handling, export structure, and Redis client usage you found."  
</div>

הסוכן יחפש קבצי middleware, יקרא את תצורת Redis שלכם, ינתח את הדפוסים שלכם, ויציע יישום שמתאים לקונבנציות המבוססות שלכם. הוראות קונקרטיות מנצחות פרומפט מופשט.

### פרומפט להסבר: טעינת קונטקסט לחלון

כשאתם שואלים "?How does X work", אתם לא בודקים את הידע של הסוכן—אתם מפעילים רצף שטוען קונטקסט לחלון לשלבים הבאים.

**המנגנון:** "Explain how our authentication middleware works" גורם לסוכן:

1. לחפש בקוד הבסיס קבצים הקשורים לאימות
2. לקרוא יישומי middleware
3. לנתח דפוסים ומבנה
4. לסנתז ממצאים

הסינתזה הזו עכשיו נמצאת בחלון הקונטקסט. כשאתם ממשיכים עם "Add rate limiting following the same pattern," לסוכן כבר יש מבנה middleware, קונבנציות טיפול בשגיאות, דפוסי export, ושימוש בתלויות טעונים בקונטקסט. הוא לא צריך לחפש שוב—הידע הרלוונטי כבר נוכח.

**שאלות הן כלי הנדסת קונטקסט.** אתם מכינים בכוונה את זיכרון העבודה של הסוכן לפני בקשת אימפלמנטציה. זה יעיל יותר מאריזת הכל לפרומפט ענקי אחד ואמין יותר מלקוות שהסוכן יחפש את הדברים הנכונים באופן אוטונומי.

:::tip ביצוע אוטונומי בטוח
שאלות בטוחות לביצוע אוטונומי—הן פעולות קריאה בלבד עם סיכון מינימלי. הגדירו את הסוכן שלכם למצב "אישור נדרש" כדי לאכוף זאת (הוא ישאל לפני ביצוע שינויים). אם ההסבר של הסוכן שגוי או לא שלם, פשוט התעלמו ממנו, שפרו את הפרומפט שלכם, ונסו שוב. רצף מעוצב היטב של שאלות חקירה הופך את משימות הקידוד האוטונומיות הבאות לאמינות הרבה יותר כי חלון הקונטקסט כבר טעון במידע מעוגן לקוד הבסיס האמיתי שלכם.
:::

### דרשו ראיות כדי לכפות עיגון

דרישת ראיות באופן מפורש מאלצת סוכנים לקרוא את הקוד האמיתי שלכם במקום לייצר ניחושים שנשמעים סבירים על בסיס דפוסים מנתוני האימון.

**המנגנון:** כשאתם דורשים "evidence (file paths, line numbers, actual values)" בפרומפט שלכם, הסוכן לא יכול לספק ראיות אלה מבלי ממש לאחזר אותן. זה הופך מה שעלול היה להיות הזיה לתגובה מעוגנת בקוד הבסיס שלכם.

**ללא דרישת ראיות:**

```
Debug the login endpoint - it's returning 500 errors
```

הסוכן עשוי להגיב: "Probably a database timeout or null pointer exception in the authentication logic."

זו השלמת דפוסים מנתוני אימון, לא ניתוח של הקוד האמיתי שלכם.

**עם דרישת ראיות:**

```
Debug the login endpoint - it's returning 500 errors.
Explain the root cause with evidence: file paths, line numbers, actual error messages.
```

עכשיו הסוכן חייב לקרוא את הקוד של ה-endpoint, לעקוב אחר הביצוע, ולצטט פרטים:
<div dir='ltr'>
"The error occurs in `src/api/auth.ts:67` where `user.profile.email` is accessed. The `profile` object is null for OAuth users (see `src/services/oauth.ts:134` - profile creation is skipped for federated auth). Stack trace shows: `TypeError: Cannot read property 'email' of null`."
</div>

**מה יכול להוות ראיות טובות:**

- **נתיבי קבצים עם מספרי שורות** - `src/auth/jwt.ts:45-67` (לא "the auth file")
- **ערכים אמיתיים מקונפיג/לוגים** - `port: 8080` (לא "a port number")
- **מזהים ספציפיים** - פונקציית `()validateJWT` (לא "the validation function")
- **הודעות שגיאה מדויקות** - stack traces מלאים או רשומות לוג (לא "an error occurred")

**שילוב עם שרשרת חשיבה (Chain-of-Thought):**

דרישת ראיות עובדת באופן עצמאי או בשילוב עם הוראות שלב-אחר-שלב. לדיבאג מורכב, השתמשו בשתיהן—[שרשרת חשיבה](../methodology/lesson-4-prompting-101.md#chain-of-thought-paving-a-clear-path) שולטת בנתיב הביצוע בעוד דרישת ראיות מבטיחה שכל שלב מעוגן:

```
Debug the failing test in UserService.test.ts:

1. Read the test file, identify which test is failing
2. Analyze test assertion: expected vs actual values
3. Trace code path through UserService to find the bug
4. Explain root cause with evidence (file paths, line numbers)
5. Propose a fix

Provide evidence for each step.
```

זה נותן לכם שליטה בביצוע (CoT) תוך אילוץ עיגון (ראיות) בכל שלב.

### שאלו שאלות הבהרה לאתגר את הלוגיקה

LLMs גרועים בלוגיקה—הם משלימים דפוסים על בסיס סבירות סטטיסטית, לא על בסיס חשיבה לוגית. כישורי ההנדסה שלכם עדיין נדרשים כדי לתפוס אי-עקביויות.

כשמשהו לא מתאים למודל המנטלי שלכם, הצביעו על זה: "If X is true, how can Y happen?"

**דוגמה:** הסוכן אומר "The config uses port 3000" אבל הלוגים שלכם מראים חיבורים על 8080. אתגרו אותו:  
<div dir='ltr'>
"You said port 3000, but logs show 8080. Explain this discrepancy with evidence from the config files and environment setup"
</div>

זה מאלץ את הסוכן לבחון מחדש את ההנחות שלו, לחפש יותר בזהירות, ולעגן את התגובה שלו בנתונים אמיתיים במקום השלמת דפוסים. השתמשו במודל המנטלי שלכם לאמת את ההיגיון של הסוכן. כשהלוגיקה לא מחזיקה, גרמו לסוכן להצדיק עם ראיות.

## תכנון מפורט: סקירה לפני ביצוע

לפני שאתם נותנים לסוכן לבצע באופן אוטונומי, סקרו את התוכנית שלו. כאן אתם תופסים אי-התאמות ארכיטקטוניות, שיקולים חסרים, ושגיאות לוגיקה—לפני שהן הופכות לקוד.

### שימו לב לאסטרטגיה והיגיון

**שאלות מפתח:**

- איך הסוכן הסיק את התוכנית הזו?
- האם העיגון היה יסודי? (האם הוא קרא קבצים רלוונטיים, בדק תיעוד, הבין אילוצים?)
- האם הוא פספס שיקולים חשובים? (אבטחה, ביצועים, תאימות לאחור, מקרי קצה?)

סקרו את ה"למה" מאחורי התוכנית, לא רק את ה"מה". אם הסוכן אומר "Implement feature X using approach Y," שאלו את עצמכם: האם הוא עיגן החלטה זו בקוד הבסיס שלכם? האם הוא שקל חלופות? האם ההיגיון מחזיק?

**דוגמה:** הסוכן מציע לשמור בקאש סשנים של המשתמששים ב-Redis עם TTL של 24 שעות. תוכנית טובה—אבל האם הוא בדק את יישום הסשנים הקיים שלכם? האם הוא שקל תאימות GDPR לנתוני סשן? האם הוא התחשב ב-cache invalidation כשמשתמשים משנים סיסמאות?

אם העיגון היה רדוד, עצרו והוסיפו קונטקסט לפני ביצוע:

```markdown
Before implementing, research our existing session management,
check for compliance requirements in COMPLIANCE.md, and propose cache
invalidation strategy for security-critical events.
Use ArguSeek, validate the approach against current security best practices.
```

### הציצו בשינויים המוצעים

לפני ביצוע אוטונומי, סקרו את התוכנית לתפיסת אי-התאמות ארכיטקטוניות ובעיות היקף. זה מחיל את [עקרונות האילוצים של שיעור 4](../methodology/lesson-4-prompting-101.md#constraints-as-guardrails) במהלך סקירת התוכנית—אתם מאמתים שהגישה המוצעת של הסוכן מוגבלת ומעוגנת מספיק לפני שהוא מבצע.

**מה אתם בודקים:**

זו התאמה ארכיטקטונית ברמה גבוהה, לא סקירת קוד שורה-אחר-שורה (אימות מגיע מאוחר יותר). אתם מבטיחים שהסוכן מעוגן בארכיטקטורה האמיתית שלכם:

- **היצמדות לדפוסים** - האם זה מתאים לדפוסים המבוססים שלנו? (אם לא, הסוכן לא היה מעוגן בקוד קיים)
- **גבולות מודולים** - האם השינויים במודולים הנכונים? (אם לא, הסוכן לא הבין ארכיטקטורה)
- **היקף מתאים** - האם הסוכן מנסה לבצע ריפקטור לחצי מקוד הבסיס כשביקשתם תיקון ממוקד?

**דוגמה:** הסוכן מתכנן להוסיף אימות כתובת דוא"ל על ידי יצירת ספריית אימות חדשה ב-`src/lib/validators/`. אבל כבר יש לכם סכמות Zod ב-`src/validation/`. זה כשל עיגון—הסוכן ייצר פתרון סביר מדפוסי אימון במקום לגלות את גישת האימות הקיימת שלכם ולהוסיף לה.

עצרו ותקנו:

```markdown
We use Zod for validation—check `src/validation/userSchema.ts` and follow that
pattern instead of creating a new library.
```

**למה זה חשוב:** תפיסת כשלי עיגון בשלב התכנון מהירה יותר משכתוב קוד שנוצר. אם התוכנית חושפת עיגון רדוד, הוסיפו אילוצים ואלצו מחקר עמוק יותר לפני ביצוע.

### שימו לב: סוכנים ממציאים במקום לעשות שימוש חוזר

**הבעיה:** כשסוכנים מתכננים יישומים, הם נוטים כברירת מחדל לייצר קוד סביר מדפוסי אימון במקום לגלות מה כבר קיים בקוד הבסיס שלכם. זו השלמת דפוסים, לא גילוי קוד קיים—הסוכן מסנתז משהו ש"נראה נכון" על בסיס מיליוני דוגמאות אימון, במקום לחפש את ה-utilities, ה-helpers, וההפשטות (abstractions) שכבר יש לכם.

מחקרים מאשרים את ההטיה הזו: קוד שנוצר על ידי AI מכיל **פי 8 יותר בלוקים משוכפלים** מקוד שנכתב על ידי בני אדם, עם ירידה במדדי קונסולידציית קוד בכל התעשייה[^1]. סוכנים ממציאים את הגלגל מחדש כי סטטיסטית קל יותר להמציא מאשר לגלות.

**דגלים אדומים בתוכניות:**

שימו לב לביטויים אלה במהלך סקירת תוכנית—הם מסמנים שהסוכן ממציא במקום מגלה:

- "Create a new utility function for..." (האם הוא חיפש כלים קיימים?)
- "Implement a helper to handle..." (האם העוזר הזה כבר קיים?)
- "Build error handling logic..." (מה עם דפוסי שגיאות קיימים?)
- "Add validation for..." (בדקו תחילה סכמות אימות קיימות)

## נקודות שחזור: רשת הביטחון שלכם

סוכנים עושים טעויות לעתים קרובות—במיוחד בזמן שאתם לומדים דפוסי עיגון ו-prompting אפקטיביים. החדשות הטובות: ככל שהכישורים שלכם משתפרים, הצורך ב-rollbacks יורד באופן דרמטי. באופן טבעי תכתבו פרומפטים טובים יותר, תתפסו בעיות במהלך סקירת תוכנית, ותנחו סוכנים ביעילות רבה יותר. אבל גם מתרגלים מנוסים מעריכים נקודות שחזור כרשת ביטחון. ההבדל בין סשן מתסכל לפרודוקטיבי מסתכם במהירות שבה אתם יכולים לבצע רולבק  כשדברים משתבשים. קידוד אייגנטי הוא הסתברותי — אתם צריכים את היכולת להחזיר לאחור גם את קונטקסט השיחה וגם שינויי קוד כשהביצוע סוטה מהכוונה שלכם.

קבעו קצב נקודות שחזור (checkpoints): צרו נקודת שחזור לפני פעולות מסוכנות, תנו לסוכן לבצע, אמתו תוצאות, ואז שמרו או בטלו. כלי קידוד AI מודרניים (Claude Code, Cursor, VS Code Copilot, וכו') כוללים יכולות נקודות שחזור מובנות שהופכות רולבק  לפשוט —זה מאפשר לכם להתנסות באגרסיביות מבלי להמר על שינויים בלתי הפיכים. אם הכלי שלכם לא תומך בנקודות שחזור ובקרה, בצעו קומיט לעתים הרבה יותר קרובות מאשר בפיתוח מסורתי: אחרי כל תוספת או שינוי מוצלח, לפני פעולות מסוכנות, לפני ששמשנים כיוון, אחרי תיקונים ידניים. זה יוצר רשת ביטחון של נקודות שחזור מאומתות שבהן כל קומיט מייצג מצב הידוע-כטוב שאתם יכולים לחזור אליו מיד. שלב האימות (מכוסה ב[שיעור 9](./lesson-9-reviewing-code.md)) קובע אם אתם שומרים או מבטלים שינויים—נקודות שחזור הופכות את ההחלטה להפיכה.

:::tip נקודות שחזור ב-Claude Code
לחצו ESC פעמיים ליצירת checkpoint ב-Claude Code. זה שומר גם קונטקסט שיחה וגם מצב קוד, ומאפשר לכם להתנסות באגרסיביות ולהחזיר מיד אם צריך.
:::

## ביצוע אוטונומי: תהליכי עבודה מקבילים והגדרת כלים

ברגע שהתוכנית נסקרה והעיגון מוצק, אתם יכולים לתת לסוכן לבצע באופן אוטונומי. עבור פיצ'רים מורכבים, ביצוע מקבילי על פני מספר מופעים של סוכנים מאיץ באופן דרמטי את הפיתוח.

### Git Worktrees: מאפשרים מקביליות אמיתית

Git worktrees מאפשרים תיקיות עבודה מרובות מריפו יחיד, כשכל אחת מכילה branch שונה. זה מאפשר הרצת מספר מופעים של סוכנים על משימות שונות במקביל, ללא קונפליקטים.

**הגדרה בסיסית:**

```bash
# Main repo in ~/project (on main branch)
git worktree add ../project-feature-auth feature/auth
git worktree add ../project-feature-api feature/api
git worktree add ../project-bugfix bugfix/login-error

# Now you have 4 separate directories:
# ~/project (main)
# ~/project-feature-auth (feature/auth branch)
# ~/project-feature-api (feature/api branch)
# ~/project-bugfix (bugfix/login-error branch)
```

**הגדרה בסיוע סוכן:**

סוכנים יכולים לעזור לייצר פקודות worktree, במיוחד כשמגדירים סביבות מקבילות מרובות. עגנו קודם עם ArguSeek לשיטות עבודה מומלצות, ואז בקשו מהסוכן לייצר פקודות.

```
Use ArguSeek to research git worktree best practices for parallel development.

Create 3 worktrees with the following specifications:
1. Authentication refactor (branch: feat/auth-refactor)
2. New analytics API (branch: feat/analytics-api)
3. Dashboard performance improvements (branch: perf/dashboard)

Output:
- Exact `git worktree add` commands for each worktree
- Recommended directory structure following best practices
```

הסוכן יחקור תהליכי worktree, יציע מבנה תיקיות נקי, ויייצר את הפקודות המדויקות. זה מהיר יותר מקריאת תיעוד ידנית ומבטיח שהפקודות מתאימות לקונטקסט הספציפי שלכם.

:::tip כלים לתהליכי עבודה מקביליים
ניהול worktrees מרובים ו-sessions של סוכנים דורש כלים יעילים. ראו **[Developer Tools](/developer-tools/terminals)** להמלצות טרמינל (Ghostty, Kitty, WezTerm) ו-**[CLI Tools](/developer-tools/cli-tools)** לכלי git וניווט קבצים מודרניים (lazygit, eza, fzf). שלבו כלי CLI ו-IDE בצורה פרגמטית—השתמשו במה שהכי יעיל לכל משימה.
:::

## נקודות מפתח

- **שאלות טוענות קונטקסט, הן לא מאמתות ידע** - "How does X work?" מפעיל רצפי חיפוש/קריאה שממלאים את חלון הקונטקסט במידע רלוונטי לשלבי ביצוע הבאים. שאלות הן כלי הנדסת קונטקסט.

- **דרשו ראיות כדי לכפות עיגון** - דרישת ראיות באופן מפורש (נתיבי קבצים, מספרי שורות, ערכים אמיתיים) מאלצת סוכנים לאחזר מידע במקום לנחש. הסוכן לא יכול לספק ראיות מבלי לקרוא את הקוד האמיתי שלכם, זה מחליף הזיות בתגובות מעוגנות. זה עובד גם באופן עצמאי או בשילוב עם שרשרת חשיבה למשימות מורכבות.

- **LLMs משלימים דפוסים, לא לוגיקה** - שיקול הדעת ההנדסי שלכם מאמת התאמה ארכיטקטונית ותופס שגיאות לוגיקה. סוכנים מטפלים בתחביר ו-boilerplate; אתם מטפלים בהיגיון ונכונות.

- **סקרו את האסטרטגיה וההיגיון של התוכנית, לא רק את הפלט** - לפני ביצוע אוטונומי, בדקו: איך התוכנית הזו נגזרה? האם העיגון היה יסודי? האם הוא פספס שיקולי אבטחה, ביצועים, או ארכיטקטורה?

- **שימו לב להמצאה על פני שימוש חוזר במהלך סקירת תוכנית** - סוכנים מעדיפים כברירת מחדל לייצר קוד סביר מדפוסי אימון במקום לגלות קוד קיים. דגלים אדומים: "create new utility," "implement helper." התערבות: אלצו גילוי קודם עם דרישת ראיות לפני מתן אפשרות ליישום.

- **צרו נקודת שחזור לפני ביצוע, קומיט אחרי אימות** - השתמשו בתכונות checkpointing מובנות כשזמינות (Claude Code, Copilot, Cursor). בלעדיהן, בצעו קומיט לעתים הרבה יותר קרובות מפיתוח מסורתי—אחרי כל שינוי מוצלח, לפני פעולות מסוכנות. סוכנים עושים טעויות תכופות; נקודות שחזור הופכות איטרציה למהירה והפיכה.

- **Git worktrees מאפשרים מקביליות אמיתית** - תיקיות עבודה מרובות מריפוזיטורי יחיד מאפשרות לכם להריץ מופעים מקבילים של סוכנים על branch-ים שונים ללא קונפליקטים. השתמשו בסוכנים לייצר פקודות הגדרת worktree מעוגנות בשיטות עבודה מומלצות.
