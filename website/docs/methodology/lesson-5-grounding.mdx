You ask your agent to fix the authentication bug. It confidently generates a solution using JWT verification patterns... that don't exist in your codebase. You use sessions, not JWTs. The agent just hallucinated a plausible implementation based on common patterns from its training data.

Here's the fundamental issue: **The agent doesn't know your codebase exists.** It doesn't know your architecture, your patterns, or your constraints. As covered in [Lesson 2: Agents Demystified](/fundamentals/lesson-2-how-agents-work), the context window is the agent's entire world—everything else doesn't exist. Without explicit grounding in your actual code and current documentation, agents generate statistically plausible solutions that may be completely wrong for your system.

**Grounding is how you inject reality into the context window.** You retrieve relevant external information—your codebase patterns, current docs, best practices—and feed it to the agent before generation. This lesson covers the engineering techniques that anchor agents in your actual system instead of hypothetical ones.

## Grounding: Anchoring Agents in Reality {#grounding-anchoring-agents-in-reality}

Grounding fails in predictable ways, so use the simplest tool that can retrieve the facts you need. Start with exact repository inspection, add semantic retrieval when keywords fail, and delegate hybrid research when the question spans too many files or sources.

## Grounding Tool 1: Agentic Search

The first grounding move is the same one you would make manually: inspect the repository. List files, search for likely terms, read the small set of files that look relevant, and follow imports or call sites from there.

When you assign "Fix the authentication bug," the agent starts with zero codebase knowledge. It doesn't know where auth code lives, what libraries you use, or how it's structured.

**Agentic search is keyword-and-file based discovery.** The agent calls tools on its own—Glob finds files matching patterns, Grep searches for keywords, Read examines code. The agent decides what to search, interprets results, and determines next steps.

In small codebases (under 10,000 lines), this works beautifully. Two or three searches return 5-10 files totaling 15,000 tokens. The agent reads them, builds a mental model, and solves the problem. Context stays clean.

## Grounding at Scale {#the-scale-problem-context-window-limits}

**At scale, agentic search creates a second problem: volume.** The agent may find the right files, but reading too many of them floods the context window. The issue is no longer missing reality — it is delivering too much reality in a form the model can reliably use.

That distinction matters:

- **Grounding problem:** the agent does not have the right facts.
- **Context engineering problem:** the agent has too many facts, poorly placed.

This lesson focuses on the grounding problem: getting the agent the right facts from the right sources. Once those facts exist, a different problem starts: deciding how much of that information should enter the context window, where it should land, and when it should be isolated or discarded. That is the context engineering problem, covered in [Lesson 6: Context Engineering](/practical-techniques/lesson-6-context-management).

## Grounding Tool 2: Semantic Search / RAG {#solution-1-semantic-search}

Agentic search works when names line up with concepts. It struggles when the code uses different language than your prompt, or when the relevant behavior is spread across files. That is where semantic search helps.

**Semantic search lets you query by meaning, not keywords.** Ask for "authentication middleware that validates user credentials" and you get relevant code even if it never mentions those exact terms.

:::tip How Semantic Search Works

**Vector embeddings:** Your code gets converted to high-dimensional vectors (768-1536 dimensions) that capture semantic meaning. Similar concepts cluster together in vector space.

**Similarity matching:** Cosine similarity finds relevant chunks. "auth middleware", "login verification", and "JWT validation" map to nearby vectors—the model understands they're semantically related even though they use different words.

**Infrastructure:** Vector databases (ChromaDB, pgvector, Qdrant) plus approximate nearest neighbor (ANN) algorithms enable fast search. Rerankers refine results. You call `code_research()`, not low-level APIs.

**Key difference:** Embedding models with vector similarity search by concept, not text. This is fundamentally different from keyword matching.
:::

**Availability depends on your tool:**

**IDE-based assistants** (Cursor, Windsurf) typically include semantic search out-of-the-box. The editor handles indexing and vector search automatically.

**CLI agents** (Claude Code, Codex, OpenCode, Pi) need MCP servers to add semantic search. The [Model Context Protocol (MCP)](https://modelcontextprotocol.io) lets you extend CLI agents with tools like semantic search, web research, and database access.

**MCP servers for semantic code search:**

- [Claude Context](https://github.com/zilliztech/claude-context) - RAG-based semantic search
- [Serena](https://github.com/oraios/serena) - LSP-based bridge (lighter weight, limited to LSP symbol scope)
- [ChunkHound](https://chunkhound.github.io) - Structured pipeline with hybrid search

Once you have semantic search (built-in or via MCP), your agent combines it with Grep for hybrid discovery: conceptual search to find the right area, precise keyword matching to locate exact implementations.

In practice, this is **retrieval-augmented generation (RAG)**: retrieve semantically relevant chunks, then generate with those chunks in context. Semantic search extends your scale to 100,000+ lines of code because you find relevant code faster with fewer false positives.

## Grounding Tool 3: Hybrid Research with Sub-Agents {#solution-2-sub-agents-for-context-isolation}

Real grounding rarely uses one technique. A good researcher mixes file listing, grep, semantic search, targeted reads, and external docs. Sub-agents package that research loop into an isolated worker.

A **sub-agent** is an agent invoked by another agent—like a function call, but for research. The orchestrator writes a prompt describing the grounding task: "Find all JWT authentication code and explain the current implementation." The sub-agent searches, reads, and analyzes in its own context. When complete, it returns a concise synthesis: "JWT implementation found at src/auth/jwt.ts using Passport.js..." The orchestrator receives the grounded facts without manually driving every search.

## Choosing the Grounding Loop {#code-grounding-choosing-tools-by-scale}

Grounding is not a code-versus-web taxonomy to memorize. It is the same judgment you already use during engineering work: if you would inspect the repository, ground in code; if you would open Google, vendor docs, changelogs, CVEs, Stack Overflow, or API references, ground on the web. Most production tasks need both.

### Freeform vs Specialized Research Agents

Research delegation comes in two shapes. **Freeform agents** explore like a developer: they follow leads, read files, search again, and adapt as they go. They are easy to use because they come built into modern coding harnesses. **Dedicated research agents** use purpose-built retrieval infrastructure — embeddings, rerankers, indexes, external search tools, graph traversal, and synthesis pipelines. That extra infrastructure usually beats freeform exploration on scale, accuracy, and speed, but it has to be installed, indexed, configured, and maintained.

**Built-in freeform agents** are generalists: Claude Code Explore/Task agents, Codex workers, Cursor/Windsurf agentic modes. They are best when the path is unclear: “find where this behavior lives,” “explain this failing test,” “compare implementation options.” They adapt well, but the result depends on the path they happened to take.

**Specialized research agents** are purpose-built retrieval systems. ChunkHound’s `code_research` and `websearch` mix exact search, semantic retrieval, traversal, targeted reads, and synthesis behind a fixed research pipeline. They are best when coverage matters: “trace auth end-to-end,” “find every module involved in rate limiting,” or “compare current OAuth2 guidance across sources.”

For small scopes, agentic search or a built-in research agent is usually enough. As the source space grows, the failure mode shifts from “can the agent search?” to “can it cover the relevant graph without drowning the parent context?” That is where specialized research agents become valuable.

## Production Pattern: Multi-Source Grounding

In production, ground the agent the same way you would investigate the problem yourself. If the answer depends on your system, inspect the code. If the answer depends on external behavior, current docs, advisories, or ecosystem practice, inspect the web. If it depends on both, use both.

This multi-source approach prevents two failure modes:

- Code-only grounding prevents hallucinations but risks outdated patterns
- Web-only grounding gives you current best practices but doesn't fit your architecture

The boundary is practical, not doctrinal. Use code grounding when the answer lives in your repo. Use web grounding when you would normally leave the repo to check docs, advisories, or ecosystem behavior. Combining both gives you solutions that work for your specific system using current standards.

## Key Takeaways

**The agent only knows what's in the context window**
Without grounding, agents hallucinate plausible solutions based on training data patterns. You ground agents by retrieving reality — your codebase, current docs, security advisories, research — before generation.

**Agentic search is the baseline**
Start with file listing, grep, targeted reads, and following references. This mirrors the first investigation loop you would run manually.

**Semantic search / RAG finds concepts when keywords fail**
Vector embeddings let you search by meaning: "authentication middleware" finds relevant code even without exact keyword matches. Retrieve the relevant chunks, then generate with those chunks in context.

**Sub-agents combine techniques and isolate the research loop**
Built-in harness sub-agents and ChunkHound's specialized research agents mix exact search, semantic search, targeted reads, and web search, then return a compact synthesis the orchestrator can use.

**Code vs web grounding is an engineering judgment**
If you would inspect the repo, use code grounding. If you would open Google, vendor docs, changelogs, CVEs, or ecosystem references, use web grounding. Most production work needs both.

**Choose the grounding loop by scale and uncertainty**
Small repositories often need only agentic search. Larger repositories benefit from semantic search and sub-agents. Large or cross-source questions need specialized research agents such as ChunkHound `code_research` and `websearch`.

**Context engineering is the next problem**
Grounding gets the facts. Context engineering keeps those facts small, timely, and visible enough to use. Lesson 6 covers the attention mechanics and runtime mechanisms behind that second problem.
