By now, you know the shape of the system: the model completes the context it receives, and the harness turns some completions into tool calls, file reads, edits, commands, and observations. Prompting is where you draw the beginning of the pattern you want the agent to continue.

A prompt is not a conversation starter. It is context design: selecting the task, inputs, constraints, examples, vocabulary, and output shape that make the intended continuation the most likely one. Good prompting narrows the completion space before the agent starts spending tool calls and editing code.

## Start With the Task and Output Target

Skip pleasantries. AI models don't need "please" or "thank you"—those tokens dilute signal without adding clarity. Start with the decision or artifact you want and the standard it must satisfy.

<PromptComparison
  goodNotes={[
    { label: 'Target', detail: 'Names the artifact, audience, and business change up front.' },
    { label: 'Scope', detail: 'Defines the systems and risks the plan must cover.' },
    { label: 'Contract', detail: 'Specifies the sections a reviewer can evaluate quickly.' },
  ]}
  bad={
    <PromptExample>
      Can you help us think through usage-based billing?<br />
      Thanks!
    </PromptExample>
  }
  good={
    <PromptExample>
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1">Produce an implementation plan for migrating workspace billing from seat-based pricing to usage-based pricing.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">Audience: engineering lead and product manager deciding whether this is ready for build planning.</span><br />
      <span className="prompt-seq-token prompt-seq-2">Cover: billing model changes, customer migration, entitlement logic, data backfill, support impact, rollout risk.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Return sections: summary, affected systems, open decisions, migration plan, rollout plan, validation checklist.</span>
    </PromptExample>
  }
/>

The technique is to name the desired artifact before supplying details. That first phrase biases the completion toward a plan, decision record, review, or spec instead of an open-ended discussion.

## Ask Questions to Load Context

Ask questions to load context. In an agent harness, a question can trigger an action sequence: search the repository, read files, query docs, inspect tool output, and synthesize the relevant findings. This is Agentic Search. The question causes the agent to load useful context before you ask it to plan or edit.

<PromptExample>
  How does the onboarding flow currently handle teammate invites before dashboard preview?
</PromptExample>

That question is enough. The agent must find the flow, inspect the relevant files, and summarize the current behavior before you ask for the change: "Make teammate invites optional before dashboard preview." Use this before planning or editing when the missing information is discoverable.

## Choose the Output Structure Early {#applying-structure-to-prompts}

Structure organizes information and directs attention. Markdown, JSON, and XML work well because they're information-dense formats heavily represented in training data. Pick the shape before details compete for attention.

<PromptAnatomy
  sequenced

    { label: 'Execution + validation', detail: 'Connects the decision to rollout, observability, and success criteria.' },
  ]}
  prompt={
    <PromptExample>
      <div className="prompt-md-section">
        <span className="prompt-md-line"><span className="prompt-md-heading-token mono-spec prompt-seq-token prompt-seq-anchor prompt-seq-1"># Task:</span> Draft an <span className="prompt-seq-token prompt-seq-1">architecture decision record</span> for moving report generation from synchronous requests to asynchronous jobs.</span>
      </div>
      <div className="prompt-md-section">
        <span className="prompt-md-heading mono-spec"><span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">## Context</span></span>
        <span className="prompt-md-line">- Enterprise reports time out during peak usage.</span>
        <span className="prompt-md-line">- Customers expect downloadable reports and email notifications.</span>
        <span className="prompt-md-line">- Support needs explainable status for delayed jobs.</span>
      </div>
      <div className="prompt-md-section">
        <span className="prompt-md-heading mono-spec"><span className="prompt-seq-token prompt-seq-2">## Constraints</span></span>
        <span className="prompt-md-line">- Preserve the existing API contract during rollout.</span>
        <span className="prompt-md-line">- Keep audit logs for every generated report.</span>
        <span className="prompt-md-line">- Support rollback without losing in-progress work.</span>
      </div>
      <div className="prompt-md-section">
        <span className="prompt-md-heading mono-spec"><span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">## Options</span></span>
        <span className="prompt-md-line"><span className="prompt-line-number">1.</span> Queue-backed async jobs</span>
        <span className="prompt-md-line"><span className="prompt-line-number">2.</span> Scheduled batch generation</span>
        <span className="prompt-md-line"><span className="prompt-line-number">3.</span> Keep synchronous flow and optimize queries</span>
      </div>
      <div className="prompt-md-section">
        <span className="prompt-md-heading mono-spec"><span className="prompt-seq-token prompt-seq-anchor prompt-seq-4">## Return</span></span>
        <span className="prompt-md-line">- Recommendation with trade-offs</span>
        <span className="prompt-md-line">- Rollout and rollback plan</span>
        <span className="prompt-md-line">- Observability gaps and validation criteria</span>
      </div>
    </PromptExample>
  }
/>

The structure makes the decision reviewable: why this matters, what cannot break, which options were considered, and how success will be verified.

## Add Constraints as Guardrails {#constraints-as-guardrails}

Without constraints, the model fills gaps with assumptions. Define boundaries explicitly.

<PromptComparison
  badLabel="Unconstrained"
  goodLabel="Constrained"
  goodNotes={[
    { label: 'Decision', detail: 'Frames the work as a migration plan instead of a vague refactor.' },
    { label: 'Boundaries', detail: 'Protects customer experience, contracts, auditability, and rollback.' },
    { label: 'Contract', detail: 'Requires evidence and open decisions instead of pretending the plan is complete.' },
  ]}
  bad={
    <PromptExample>
      Move file processing to background jobs.
    </PromptExample>
  }
  good={
    <PromptExample>
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1">Design a migration plan for moving file processing from synchronous API requests to asynchronous jobs.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">Constraints:</span><br />
      <span className="prompt-seq-token prompt-seq-2">- Preserve the current API response contract during rollout.</span><br />
      <span className="prompt-seq-token prompt-seq-2">- No user-visible regression for upload progress or error recovery.</span><br />
      <span className="prompt-seq-token prompt-seq-2">- Preserve audit logs and idempotency guarantees.</span><br />
      <span className="prompt-seq-token prompt-seq-2">- No schema change without a backfill and rollback plan.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Return affected systems, migration phases, risks, evidence needed, and decisions that require human approval.</span>
    </PromptExample>
  }
/>

The unconstrained version invites the agent to jump into implementation. The constrained version defines the operational envelope before code exists.

## Use Examples When Rules Are Hard to Describe

Few-shot prompting gives the model input/output examples before the real task. Instead of describing every rule, you show the pattern you want completed.

<PromptComparison
  badLabel="No examples"
  goodLabel="Few-shot"
  goodNotes={[
    { label: 'Pattern', detail: 'Examples define the input and output shape the model should continue.' },
    { label: 'Boundaries', detail: 'Categories show fuzzy product judgment better than prose alone.' },
    { label: 'Continuation', detail: 'The final unfinished Output starts the exact completion pattern.' },
  ]}
  bad={
    <PromptExample>
      Classify this onboarding complaint:<br />
      <br />
      "I wanted to try the dashboard, but it asked me to invite teammates before I knew what the product did."
    </PromptExample>
  }
  good={
    <PromptExample>
      Classify onboarding friction reports into one primary category: Copy, Flow, Permissions, Data model, Reliability.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1">Examples:</span><br />
      <span className="prompt-seq-token prompt-seq-1">Input:</span> "I don't understand what a workspace is."<br />
      <span className="prompt-seq-token prompt-seq-1">Output:</span> <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">Copy</span> — terminology blocks comprehension.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-1">Input:</span> "It asks me to invite teammates before I see value."<br />
      <span className="prompt-seq-token prompt-seq-1">Output:</span> <span className="prompt-seq-token prompt-seq-2">Flow</span> — sequence creates premature commitment.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-1">Input:</span> "I can't connect the billing data source without admin access."<br />
      <span className="prompt-seq-token prompt-seq-1">Output:</span> <span className="prompt-seq-token prompt-seq-2">Permissions</span> — access boundary blocks activation.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Now classify:</span><br />
      Input: "I wanted to try the dashboard, but it asked me to invite teammates before I knew what the product did."<br />
      <span className="prompt-seq-token prompt-seq-3">Output:</span>
    </PromptExample>
  }
/>

Start with one example when the pattern is simple. Add more only when the output stays inconsistent, and prefer diverse examples that cover real boundary cases.

## Reverse Prompt From an Ideal Output

When you already have an ideal output, don't guess the prompt from scratch. Give that output to the model and ask it to infer the instructions, constraints, structure, and validation criteria that would reproduce it.

<PromptAnatomy
  sequenced
  items={[
    { label: 'Ideal output', detail: 'Anchors the model on the target artifact before asking it to infer anything.' },
    { label: 'Inference', detail: 'Extracts the audience, sections, evidence rules, tone, and operating constraints behind the artifact.' },
    { label: 'Template', detail: 'Converts the inferred pattern into a reusable forward prompt with variables.' },
    { label: 'Validation', detail: 'Closes the loop by testing whether the new prompt reproduces the target style.' },
  ]}
  prompt={
    <PromptExample>
      Here is an example of the <code className="prompt-seq-token prompt-seq-anchor prompt-seq-1">architecture decision record</code> I want from an AI coding assistant:<br />
      <br />
      [Paste one complete <span className="prompt-seq-token prompt-seq-1">decision record</span> whose structure and quality you want to reuse.]<br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">Reverse-engineer</span> the prompt that would produce this kind of decision record.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-2">Infer</span>:<br />
      - intended audience and decision scope<br />
      - required sections<br />
      - <span className="prompt-seq-token prompt-seq-2">evidence requirements</span><br />
      - expected trade-off language<br />
      - <span className="prompt-seq-token prompt-seq-2">constraints</span> the reviewer must preserve<br />
      <br />
      Return:<br />
      <span className="prompt-line-number">1.</span> A <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">reusable prompt template</span><br />
      <span className="prompt-line-number">2.</span> The <span className="prompt-seq-token prompt-seq-3">variables</span> I should fill in for each decision<br />
      <span className="prompt-line-number">3.</span> A <span className="prompt-seq-token prompt-seq-anchor prompt-seq-4">validation checklist</span> for testing whether the prompt reproduces this decision quality
    </PromptExample>
  }
/>

Use multiple examples when the target output is complex. A single example helps you move fast, but several examples make the shared structure clearer and reduce invented constraints.

## Add Persona When Vocabulary Matters

Use personas when domain-specific terminology matters: security, performance, accessibility, reliability. Skip personas when the task is straightforward and adding persona context wastes tokens without adding value.

<PromptComparison
  badLabel="Generic"
  goodLabel="Reliability-focused"
  goodNotes={[
    { label: 'Persona', detail: 'Sets reliability vocabulary: blast radius, failover, SLOs, runbooks.' },
    { label: 'Review targets', detail: 'Names exact operational risks instead of leaving “issues” undefined.' },
    { label: 'Assumptions', detail: 'Forces production posture: partial outages, human response, rollback.' },
  ]}
  bad={
    <PromptExample>
      Review this multi-region design for issues.
    </PromptExample>
  }
  good={
    <PromptExample>
      <code className="mono-human prompt-seq-token prompt-seq-anchor prompt-seq-1">You are a staff-level reliability engineer reviewing a proposed multi-region failover design.</code><br />
      <br />
      Review for:<br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">blast radius, failover trigger ambiguity, data consistency risk, observability gaps, runbook gaps, rollback assumptions.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Assume partial regional outages, degraded dependencies, and an on-call engineer executing under time pressure.</span>
    </PromptExample>
  }
/>

The persona doesn't add knowledge. It changes which knowledge gets retrieved by shifting vocabulary. The same control interface applies to code search, web research, vector databases, and sub-agent delegation: "multi-region failover risks" retrieves different context than "deployment issues."

## Define the Reasoning Path for Multi-Step Work {#chain-of-thought-paving-a-clear-path}

When tasks require multiple steps, define the execution path. You are not asking for hidden reasoning; you are dictating the route so the agent cannot skip diagnosis, evidence, or validation.

<PromptComparison
  badLabel="Without path"
  goodLabel="With path"
  goodNotes={[
    { label: 'Scope', detail: 'Pins the investigation to one business regression.' },
    { label: 'Path', detail: 'Forces diagnosis through funnel data, product changes, hypotheses, and fixes.' },
    { label: 'Evidence', detail: 'Separates measured facts from plausible explanations.' },
  ]}
  bad={
    <PromptExample>
      Figure out why enterprise onboarding completion dropped.
    </PromptExample>
  }
  good={
    <PromptExample>
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1">Investigate why enterprise onboarding completion dropped after the workspace invitation redesign.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2"><span className="prompt-line-number">1.</span> Identify the affected funnel stages and the before/after metric deltas.</span><br />
      <span className="prompt-seq-token prompt-seq-2"><span className="prompt-line-number">2.</span> Compare the old and new onboarding sequence at the product-experience level.</span><br />
      <span className="prompt-seq-token prompt-seq-2"><span className="prompt-line-number">3.</span> List hypotheses across copy, flow, permissions, data readiness, and reliability.</span><br />
      <span className="prompt-seq-token prompt-seq-2"><span className="prompt-line-number">4.</span> Mark each hypothesis as supported, unsupported, or needs data.</span><br />
      <span className="prompt-seq-token prompt-seq-2"><span className="prompt-line-number">5.</span> Propose experiments or fixes with expected signal.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Provide evidence for every claim and call out assumptions separately.</span>
    </PromptExample>
  }
/>

Use this whenever the path must be enforced: diagnosis before recommendation, evidence before conclusion, validation before implementation. It does not make the agent deterministic; it just makes skipped steps and premature claims easier to detect.

## Require Evidence for Claims {#require-evidence-to-force-retrieval}

Evidence requirements control what the model can claim. When the agent explains a plan, discrepancy, or bug, require file paths, line numbers, concrete values, exact error messages, cited sources, metric names, timestamps, or customer quotes.

<PromptComparison
  badLabel="Weak claim—pattern completion"
  goodLabel="Grounded claim—retrieval with evidence"
  bad={
    <PromptExample>
      The onboarding redesign probably failed because inviting teammates too early added friction.
    </PromptExample>
  }
  good={
    <PromptExample>
      The largest drop is at <code>workspace_invite_shown → first_dashboard_view</code>: 62% to 41% in <code>activation_funnel_enterprise</code> for May 14–21. Three support tickets quote confusion about mandatory invites, and the redesign spec lists teammate invitation before dashboard preview.
    </PromptExample>
  }
/>

The second claim is verifiable. If the metric, date range, quote, or spec reference is wrong, you spot the drift immediately. Evidence requirements do not make the model deterministic; they make unsupported claims visible.

## Replace Negation With Positive Instructions

Negation is a weak control surface because the model is not applying symbolic rules. It predicts continuations from the whole context, and the forbidden concept is still part of that context. Use "do not" to mark the boundary, then immediately specify the desired behavior in positive, concrete terms.

<PromptComparison
  badLabel="Risky"
  goodLabel="Better—positive target plus boundary"
  goodNotes={[
    { label: 'Negation', detail: 'Names the unsafe product outcome explicitly.' },
    { label: 'Positive opposite', detail: 'Replaces vague avoidance with a concrete flow principle.' },
    { label: 'Mechanism', detail: 'Names sequencing rules that preserve the intended experience.' },
  ]}
  bad={
    <PromptExample>
      Improve onboarding.<br />
      <code className="mono-spec">Do NOT</code> make the flow confusing.
    </PromptExample>
  }
  good={
    <PromptExample>
      Improve first-run onboarding.<br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1"><code className="mono-spec">Do NOT</code> introduce steps that make activation feel blocked.</span><br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2"><code className="mono-spec">Instead</code>, keep the first-run path linear and value-first.</span><br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Ask for workspace name before teammate invites; show optional setup after dashboard preview; label optional steps as skippable.</span>
    </PromptExample>
  }
/>

The safer pattern states the forbidden outcome, immediately supplies the desired experience, then names the sequencing mechanism. It gives the model a concrete continuation to follow instead of leaving the forbidden concept as the strongest signal.

## Delegate Math to Executable Code

Math needs exact execution, not plausible continuation. Because LLMs generate probable next tokens rather than verified results, they can produce arithmetic-shaped answers that look coherent but are wrong. Ask them to write executable code for arithmetic, measurement, parsing, and validation.

<PromptComparison
  badLabel="Don't rely on LLMs for math"
  goodLabel="Instead, have the model write code"
  goodNotes={[
    { label: 'Executable', detail: 'Moves pricing arithmetic into code instead of asking for guessed impact.' },
    { label: 'Inputs', detail: 'Defines every variable and cohort rule the calculation depends on.' },
    { label: 'Verification', detail: 'Requires sensitivity checks and tests so assumptions are visible.' },
  ]}
  bad={
    <PromptExample>
      Estimate whether usage-based pricing will increase revenue without upsetting enterprise customers.
    </PromptExample>
  }
  good={
    <PromptExample>
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-1">Write a script that models the impact of usage-based pricing across customer cohorts.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-2">Inputs:</span><br />
      <span className="prompt-seq-token prompt-seq-2">- current MRR by account</span><br />
      <span className="prompt-seq-token prompt-seq-2">- monthly event volume by account</span><br />
      <span className="prompt-seq-token prompt-seq-2">- proposed unit price, included volume, caps, and discounts</span><br />
      <span className="prompt-seq-token prompt-seq-2">- account segment and contract tier</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-1">Return cohort revenue deltas, accounts with &gt;20% increase, and a sensitivity table.</span><br />
      <br />
      <span className="prompt-seq-token prompt-seq-anchor prompt-seq-3">Include tests for pricing formulas, caps, discounts, and missing data handling.</span>
    </PromptExample>
  }
/>

The model may generate plausible-sounding business impact because probability can imitate correctness without guaranteeing it. It is far better at writing code that computes the result, tests the formulas, and exposes the assumptions behind the estimate.

## From Prompt Controls to Workflow

These controls make one prompt reviewable: target artifact, separated inputs, constraints, examples, evidence, positive instructions, and executable validation. They are the smallest unit of agent operation.

For production work, the next problem is orchestration. A large change is not one perfect prompt; it is a sequence of prompts that research the system, produce a plan, execute within constraints, and validate the result.
