In Lesson 1, we reduced the model to its operating core: context in, predicted tokens out. A model can emit prose, code, a reasoning trace, or a tool-call-shaped message, but it is still predicting tokens from the supplied context.

An agent is the software system wrapped around that model call. The harness supplies context, exposes tools, parses structured outputs, executes approved actions, appends observations, and decides whether to continue. The LLM is the probabilistic engine; the agent is the control system that turns its outputs into useful work.

## The Minimal Agent Loop

The minimal agent loop is not a cognitive cycle. It is an application loop around repeated LLM calls. Each pass prepares input context, gets model output, validates the output, performs any approved side effect, and serializes the result back into the next context.

  <figcaption>
    The harness turns model output into a controlled loop: context in, validated action out, observation back into context.
  </figcaption>
</figure>

Once you see the loop as context flow, the agent boundary becomes explicit. The model only predicts the next token sequence. The harness owns validation, execution, observation capture, stopping conditions, and budget control.

A minimal harness looks like normal application code:

```ts
const observations = [];

while (!done) {
  const context = assembleContext({ task, memory, observations, tools });
  const modelOutput = await llm.complete(context);
  const action = parseAndValidate(modelOutput, { tools, policy });

  if (action.type === 'final') return action.response;

  const observation = await executeOutsideModel(action);
  observations.push(serializeObservation(observation));
  done = shouldStopOrIterate({ task, observations, budget });
}
```

There is no separate agent substance in this loop. The leverage comes from context assembly, output parsing, execution policy, and iteration.

## Everything Useful Becomes Context

Agent behavior is shaped by the tokens available at the next model call. The model's statelessness follows from Lesson 1: each call sees only the supplied context. Conversation history, previous tool results, repo snippets, and summaries are not memories inside the model; they are input tokens selected by the agent system.

That is why context engineering is the central operator skill. You are not programming a mind. You are shaping the input stream to a probabilistic dependency.

### The Stateless Advantage

Statelessness is useful because a fresh context can produce an independent review, alternate design, or clean re-run without defending earlier conversation drift. The advantage comes from software control over context, not from the model recalling anything.

### What the Context Stream Looks Like

When you ask an agent to "Add email validation to the registration endpoint," the harness builds one continuous interaction stream: system instructions, your request, tool calls, tool results, and final response.

That stream may be represented as provider-native messages rather than one literal string, and some providers hide parts of reasoning behind summarized "thinking" outputs. The operating model stays the same: the next LLM call is conditioned on the context the harness provides.

## Memory Is Retrieval, Not Recall

Agent "memory" is software-managed context selection. The LLM does not remember your project conventions, previous sessions, or domain facts unless the agent retrieves or stores them and feeds them into a later call.

Common memory layers include:

- **Working memory:** the current conversation, active files, recent tool observations, and task state.
- **Persistent rules:** `CLAUDE.md`, `AGENTS.md`, project instructions, style guides, and saved preferences.
- **Summaries:** compressed session state that keeps long work from exceeding the context budget.
- **RAG and search:** keyword search, vector databases, hybrid retrieval, knowledge graphs, and documentation indexes.
- **External records:** tickets, design docs, audit logs, prior decisions, and production telemetry.

All of them solve the same problem: choose which tokens deserve space in the next request. Good memory systems retrieve precise, current, source-backed context. Bad memory systems flood the model with stale summaries, irrelevant vector hits, or unverified preferences.

In code, memory is usually just another input to context assembly:

```ts
const context = [
  systemPrompt,
  await readFile('AGENTS.md'),
  await retrieveRelevantDocs(task),
  await summarizePreviousSession(task),
  formatRecentObservations(observations),
  userRequest,
];
```

The quality question is not "does the model remember?" It is "did the software retrieve the right evidence and compress it without losing the constraint that matters?"

## Tools Are Structured Requests

Tools are functions, commands, or API operations exposed by the harness. The model requests them through structured output; the harness validates and executes them.

A coding harness typically exposes operations like:

- **Search:** `grep`, semantic search, symbol lookup, documentation lookup
- **Read:** files, diffs, logs, issues, runtime state
- **Modify:** edit, write, refactor, apply patch
- **Execute:** tests, linters, builds, scripts, shell commands
- **Integrate:** APIs, databases, cloud services, browsers, internal tools

The failure boundary is also in the harness. It must reject malformed calls, enforce permissions, redact secrets, cap output, surface errors, and ask for approval when an action is risky. The model can suggest an action; the agent system owns execution.

A model might emit a structured request like this:

```json
{
  "tool": "bash",
  "args": {
    "command": "npm test -- --runInBand"
  }
}
```

The harness then applies policy before the side effect:

```ts
const request = parseToolCall(modelOutput);
assertAllowed(request, policy);

const result = await runTool(request.tool, request.args);
observations.push(serializeObservation(result));
```

The JSON did not run tests. The harness did.

## Integration: Prefer CLIs, Use MCP When It Fits

For developer agents, mature CLIs are often the best tool interface. Models have seen common CLIs in training data, command output is naturally textual, shell composition is powerful, and files/pipes give simple interoperability without spending context on large tool schemas.

MCP still matters, but it should be right-sized. Use MCP when you need remote service integration, OAuth or multi-tenant auth, governed access, auditability, standardized discovery, or a service that has no good CLI. Do not treat MCP as the default path for every local developer workflow.

:::tip[Practical default]
Start with the smallest reliable interface the model already understands. For local code work, that is usually files plus commands. Add MCP when auth, governance, remote state, or missing CLIs justify the extra surface area.
:::

## Two Agent Architectures

The useful distinction is not CLI versus IDE versus chat UI. Those are interaction surfaces. The architectural distinction is who owns the control plane.

### 1. Interactive Harness Agents

Interactive coding agents — Claude Code, Codex-style terminals, Pi, OpenClaw, Cursor/Copilot-style coding agents — let the LLM drive many next-step decisions through tool calls or command requests. The harness provides guardrails and execution, but the model often chooses the next search, read, edit, test, or question.

They are strong when the work is open-ended:

- exploring an unfamiliar codebase
- implementing a feature with unclear local constraints
- debugging through iterative evidence gathering
- refactoring with human oversight
- coordinating several investigative branches

Their main dependency is operator skill. Good prompts, scoped tasks, rules files, verification commands, and review loops dramatically improve outcomes. Weak context produces wandering behavior, excessive tool use, and plausible but wrong conclusions.

### 2. Structured Control-Plane Systems

Structured agent systems put deterministic software in charge. The workflow, state transitions, routing, budgets, retries, validation, audit logs, and stop conditions live in code. LLM calls happen at bounded junctions where language understanding or generation is useful.

Common patterns include:

- search/research pipelines that gather sources, rank evidence, and ask the LLM to synthesize
- extraction/classification/routing systems with schema validation
- Temporal activities or durable workflows that call models inside explicit steps
- LangGraph-style state graphs with typed transitions
- CrewAI Flows or similar orchestration where code owns the process

These systems are stronger when reliability, repeatability, cost control, and auditability matter more than broad autonomy. The model becomes a bounded worker inside a larger deterministic machine.

A structured research pipeline, for example, should not let the model decide every step:

```ts
const sources = await retrieveSources(query);
const rankedSources = rankByRelevanceAndTrust(sources);
const draft = await llm.complete(synthesisPrompt(query, rankedSources));

return validateCitations(draft, rankedSources);
```

The LLM handles synthesis. Deterministic code owns retrieval, ranking, citation validation, and the decision to reject an unsupported answer.

## Tradeoffs

| Dimension | Interactive harness agents | Structured control-plane systems |
| --- | --- | --- |
| Best for | Ambiguous coding work, exploration, debugging, implementation with oversight | Repeatable workflows, production pipelines, governed operations |
| Control plane | LLM chooses many next actions; harness enforces boundaries | Code owns workflow, state, routing, validation, and budgets |
| Strength | Flexibility, breadth, fast adaptation, high operator leverage | Reliability, auditability, predictable cost, easier productization |
| Main risk | Drift, over-tooling, missed context, inconsistent judgment | Rigidity, upfront design cost, narrow task coverage |
| Operator role | Steer, scope, review, approve, verify | Design workflow, define schemas, set policy, monitor metrics |
| Validation | Often interactive: tests, reviews, diffs, approvals | Built in: typed outputs, gates, retries, audits, deterministic checks |

Use interactive harness agents when discovery and adaptation dominate. Use structured control planes when the workflow is known and the cost of a bad autonomous step is high.

## Why This Matters for the Rest of the Book

The rest of this course is about operating the control surface around the model: grounding, context selection, task decomposition, verification, parallel work, and reliability levers. Those practices apply whether the interface is an interactive coding harness or a production workflow with bounded LLM calls.

The agent is not magic. It is a token predictor embedded in software that controls context, tools, memory, and validation. Once you see that boundary clearly, agent operation becomes engineering.
